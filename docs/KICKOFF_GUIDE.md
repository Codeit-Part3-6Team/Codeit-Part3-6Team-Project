# 프로젝트 킥오프 가이드

이 문서는 팀원이 프로젝트 첫날에 전체 흐름을 이해하고, 각자 무엇을 준비해야 하는지 알 수 있도록 만든 안내 문서입니다.
개발 경험이 많지 않아도 이해할 수 있게 설명하되, 실제 작업에 필요한 기준과 산출물도 함께 정리합니다.

## 1. 우리가 먼저 맞추려는 것

팀 프로젝트에서 가장 힘든 순간은 보통 마지막입니다.
각자 열심히 만들었는데 데이터 형식이 다르거나, 모델이 받는 입력이 다르거나, 결과 파일 위치가 달라서 합치기 어려워질 수 있습니다.

이번에는 그런 일을 줄이기 위해 처음부터 아래 네 가지를 맞춥니다.

- 어떤 데이터를 사용할지
- 각자 어떤 결과물을 남길지
- 실험 결과를 어디에 기록할지
- GitHub에서 어떻게 작업을 주고받을지

한 문장으로 말하면, **각자 만든 작업물이 마지막에 자연스럽게 붙도록 미리 약속을 정하는 것**입니다.

## 2. 전체 작업 흐름

```text
데이터 준비
-> 데이터 형식 확인
-> 모델 학습
-> 결과 저장
-> 데모 앱 연결
-> 보고서/발표 정리
```

텍스트 분류 프로젝트라면 이런 흐름입니다.

```text
문장 데이터 준비
-> text, label 형태로 정리
-> 학습 코드 실행
-> 성능과 모델 파일 저장
-> 앱에서 문장을 넣으면 예측 결과 표시
-> 실험 결과를 보고서에 정리
```

이 흐름에서 중요한 것은 “누가 어느 단계의 결과물을 다음 사람에게 넘기는가”입니다.

```text
Data 담당 -> Experiment 담당: 정리된 데이터와 데이터 설명
Experiment 담당 -> App 담당: 모델 파일과 입력/출력 형식
Experiment 담당 -> Report 담당: 실험 결과와 최종 모델 선택 이유
App 담당 -> Report 담당: 데모 화면과 사용 흐름
PM -> 전체: 일정, 기준, 통합 상태
```

## 3. 역할은 “담당 구역”입니다

역할은 일을 제한하기 위한 것이 아닙니다.
누가 어떤 부분을 책임지고 정리할지 정하는 기준입니다.
필요하면 서로 도와도 됩니다.

| 역할 | 쉽게 말하면 | 맡는 일 | 반드시 남길 것 |
|---|---|---|---|
| PM | 전체 흐름을 정리하고 연결하는 사람 | 일정, 기준, 역할 조율, 통합 관리 | 운영 기준, 일정, 결정사항, 통합 체크 |
| Data Engineer | 데이터를 모델이 쓸 수 있게 정리하는 사람 | 데이터 정리, 데이터 품질 확인, 데이터 설명 | train/valid/test, class_map, dataset_info, 데이터 이슈 |
| Experiment Lead | 모델을 고르고 실험하는 사람 | 모델 선택, 학습 실행, 결과 비교 | config, metrics, 모델 파일, 실험 해석 |
| Application Engineer | 결과를 사용자가 볼 수 있게 만드는 사람 | 데모 화면, API, 입력/출력 연결 | 실행 가능한 데모, 사용 방법, 화면 캡처 |
| Presentation Lead | 프로젝트를 설명 가능한 이야기로 만드는 사람 | 보고서, 발표자료, 발표 흐름 정리 | 최종 보고서, 발표자료, 발표 흐름, 리허설 체크 |

중요한 점:

- 발표 담당자가 모든 문서를 혼자 쓰는 것은 아닙니다.
- 각 담당자가 자기 파트의 설명과 근거를 남깁니다.
- 발표/보고서 담당자는 그 자료를 모아 하나의 흐름으로 정리합니다.
- 역할별 업무량이 달라질 수 있으므로 PM이 중간에 조정합니다.

5명 팀에서는 아래 두 가지 방식 중 하나를 선택합니다.

```text
A안: 웹앱/데모 시도
PM / Data Engineer / Experiment Lead / Application Engineer / Presentation Lead

B안: 모델/실험 강화
PM / Data Engineer / Model Engineer / Experiment Lead / Presentation Lead
```

Application Engineer를 맡고 싶은 팀원이 있으면 A안으로 갑니다.
웹앱 담당이 없거나 일정이 촉박하면 B안으로 가고, 데모는 Colab Notebook과 결과 시각화 중심으로 낮춥니다.

## 4. 데이터 약속표

우리는 데이터를 아무 형태로나 넘기지 않고, 모두가 이해할 수 있는 공통 형태로 맞춥니다.
이 공통 형태를 여기서는 **Data Contract**, 쉽게 말해 **데이터 약속표**라고 부릅니다.

텍스트 분류라면 가장 기본 형태는 이렇습니다.

```csv
text,label
이 영화는 재미있다,positive
이 앱은 자주 멈춘다,negative
```

이미지 분류라면 이런 형태입니다.

```csv
image_path,label
images/img_000001.jpg,cat
images/img_000002.jpg,dog
```

데이터 담당자가 준비할 것:

- `train.csv`: 학습용 데이터
- `valid.csv`: 학습 중 성능 확인용 데이터
- `test.csv`: 최종 확인용 데이터
- `class_map.json`: label 이름과 번호의 대응표
- `dataset_info.json`: 데이터 버전, 클래스 수, split 개수, 주의사항

학습 전에 확인할 것:

- 필요한 컬럼이 있는가
- label 이름이 정리되어 있는가
- 빈 값이 없는가
- 이미지라면 파일 경로가 실제로 존재하는가
- train/valid/test가 섞이지 않았는가

이 확인을 자동으로 해주는 것이 `validate_data.py`입니다.

## 5. 전처리와 증강은 구분합니다

전처리는 모두 같은 의미가 아닙니다.
이번 프로젝트에서는 아래처럼 구분합니다.

| 구분 | 목적 | 예시 | 적용 위치 |
|---|---|---|---|
| 데이터 정리 | 데이터가 파이프라인에 들어가게 만들기 | label 정리, split 생성, 결측 확인 | 데이터 준비 단계 |
| 입력 변환 | 모델이 받을 수 있는 형태로 바꾸기 | tokenization, resize, normalize | 학습/예측 코드 |
| 학습용 증강 | 성능을 높이기 위한 변형 | random crop, noise, synonym replace | 보통 train에만 |

기본 원칙:

- 공통 데이터는 너무 많이 바꾸지 않습니다.
- 학습 성능을 위한 증강은 실험 설정으로 관리합니다.
- 검증/테스트/실제 예측에는 랜덤 증강을 적용하지 않습니다.

## 6. 실험 기록은 왜 남기나

모델 실험을 하다 보면 “어떤 설정으로 돌렸더라?”를 잊기 쉽습니다.
그래서 실험마다 아래 내용을 남깁니다.

- 어떤 데이터로 실행했는지
- 어떤 모델을 썼는지
- 어떤 설정으로 실행했는지
- 결과가 어땠는지
- 모델 파일이 어디 저장됐는지
- 실패했다면 왜 실패했는지

이렇게 남겨두면 좋은 점:

- 잘 된 실험을 다시 재현할 수 있습니다.
- 실패한 실험을 반복하지 않을 수 있습니다.
- 보고서에 근거를 넣기 쉬워집니다.
- 팀원이 내 실험을 이어받기 쉬워집니다.

실험 결과 폴더에는 앞으로 이런 파일들이 남게 됩니다.

```text
config.yaml
metrics.json
history.csv
run_info.json
best_model
README.md
```

## 7. GitHub는 작업 게시판입니다

GitHub는 코드 저장소이기도 하지만, 이 프로젝트에서는 **작업 게시판**처럼 사용합니다.

기본 흐름:

```text
Issue에서 할 일 확인
-> 자기 작업 진행
-> PR로 변경 내용 공유
-> 팀원이 확인
-> 통합
```

Issue에는 “무엇을 해야 하는지”를 적습니다.
PR에는 “무엇을 바꿨고 어떻게 확인했는지”를 적습니다.

PR에 꼭 적을 것:

- 작업 내용
- 변경 이유
- 실행해서 확인한 것
- 관련 산출물 경로
- 리뷰가 필요한 부분

처음부터 Git 명령어를 완벽히 외울 필요는 없습니다.
중요한 것은 내 작업과 결과를 팀원이 확인할 수 있게 남기는 것입니다.

## 8. 우리가 쓸 폴더를 쉽게 보면

```text
data/       데이터
configs/    실험 설정
src/        실제 코드
app/        데모 앱
docs/       설명 문서
reports/    실험 기록과 보고서 자료
tests/      자동 확인 코드
```

조금 더 풀어 말하면:

- `data/`: 데이터가 들어가는 곳
- `configs/`: 실험 조건을 적어두는 곳
- `src/`: 학습, 예측, 검증 코드가 있는 곳
- `app/`: 사용자에게 보여줄 데모를 만드는 곳
- `docs/`: 규칙과 설명을 정리하는 곳
- `reports/`: 실험 결과와 발표 자료를 모으는 곳
- `tests/`: 기본 기능이 깨지지 않았는지 확인하는 곳

실험을 실행하면 `experiments/` 폴더에 실험별 결과가 한 덩어리로 저장됩니다.

## 9. 첫날에 할 일

첫날에는 무언가를 많이 만들기보다, 모두가 같은 출발선에 서는 것이 중요합니다.

체크리스트:

- 프로젝트 저장소 받기
- conda 환경 켜기
- 테스트 실행하기
- 데이터 검증 명령 한 번 실행하기
- 자기 역할 확인하기
- 첫 작업 Issue 확인하기
- 막힌 점을 어디에 남길지 확인하기

환경 확인 명령:

```bash
conda activate codeit-ml-pipeline
pytest
```

텍스트 데이터 검증 예시:

```bash
python scripts/run_validate.py --data-dir data/text_processed
```

텍스트 smoke test 예시:

```bash
python scripts/run_train.py --config configs/smoke_test_text.yaml --project-root .
python scripts/run_predict.py --config configs/smoke_test_text.yaml --project-root . --input data/text_processed/sample_positive.txt
```

## 10. Daily Report는 짧게 씁니다

Daily Report는 긴 보고서가 아닙니다.
아래 세 가지만 남기면 됩니다.

- 오늘 한 일
- 관련 링크
- 막힌 점

예시:

```text
오늘 한 일:
- text_processed 데이터 초안 작성
- validate_data.py 실행 확인

관련 링크:
- Issue #3
- PR #5

막힌 점:
- label 기준을 positive/negative로 갈지, 0/1로 갈지 확인 필요
```

막힌 점은 숨기는 것이 아니라 빨리 공유하는 것이 좋습니다.
그래야 팀 전체 속도가 빨라집니다.

## 11. 오늘 기억할 것

이 프로젝트의 핵심은 “처음부터 완벽하게 만들기”가 아닙니다.

핵심은 이것입니다.

```text
각자 맡은 일을 하되,
결과물이 마지막에 잘 붙도록
데이터 형식, 실험 기록, GitHub 흐름을 맞춘다.
```

처음에는 이 기준만 지켜도 충분합니다.
세부 규칙은 프로젝트가 진행되면서 필요한 만큼 추가합니다.
