# RFP 분석 서비스 기획안

회의 발표용. 각 안건을 아래 네 가지로 정리했다:

1. **시나리오** — 이 서비스는 뭘 하는가
2. **필요한 Tool** — 무슨 기능이 필요한가
3. **수정 범위 + 소요 기간** — 기존 코드 얼마나 고치고, 얼마나 걸리는가
4. **데이터 준비** — 어떤 데이터가 필요하고, 어떻게 마련하는가

---

## S — RFP 단건 Q&A (현재 완료)

### 시나리오

사용자가 질문하면 RFP 문서에서 답을 찾아준다.

```
Q: "예산이 얼마야?"
A: "5천만 원입니다." (출처: 문서 3페이지)
```

### Tool

따로 없다. 지금 파이프라인의 retriever + answerer가 그대로 하나의 기능으로 동작한다.

### 수정 범위

없음. 완료.

### 데이터

- `data_list.csv` (문서 100건) — 준비 완료
- `eval_questions.csv` (897개 평가 질문) — 준비 완료

## M — RFP 자동 요약 리포트

### 시나리오

RFP 파일 하나를 던지면, 예산·일정·자격·서류·발주기관·평가기준을 한 장짜리 표로 정리해준다. PM 입장에서는 "100페이지짜리 RFP를 읽는 대신 요약본을 본다"에 해당한다.

```
입력: RFP 파일 1개

출력:
┌──────────┬──────────────────────────┐
│ 사업명   │ OOO 시스템 구축           │
│ 예산     │ 5억 원                    │
│ 기간     │ 계약일로부터 6개월         │
│ 발주기관 │ 조달청                    │
│ 자격     │ SW사업자, 최근 3년 실적    │
│ 제출서류 │ 제안서, 사업자등록증 외 6종 │
│ 평가기준 │ 기술 80%, 가격 20%         │
└──────────┴──────────────────────────┘
```

### 필요한 Tool

| Tool | 하는 일 |
|------|---------|
| `extract_info` | 문서에서 "예산", "일정", "자격" 등 항목별로 값을 뽑는다 |
| `check_completeness` | 빠진 항목이 없는지 확인한다 ("자격 정보가 문서에 없음" 표시) |
| `format_report` | 추출 결과를 마크다운 표로 깔끔하게 정리한다 |

3개 모두 answerer(LLM 호출부)를 재활용한다. 단, 자유 텍스트가 아니라 "예산: 5억, 기간: 6개월" 같은 구조화된 출력이 필요하므로 answerer에 StructuredOutput 기능을 추가해야 한다.

### 수정 범위 + 소요 기간

```
수정: src/rag/engines/langchain.py   1~2일
      → answerer가 Pydantic 스키마를 받아 구조화된 dict를 반환하게 변경

수정: src/rag/pipeline.py             1일
      → run_rag_summarize() 함수 추가. 내부에서 위 3개 Tool 순차 호출

추가: configs/experiments/rag/rag_summary.yaml   반나절
      → 기존 rag_langchain.yaml 복사 후 프롬프트만 교체
```

**총 소요: 1~2주** (코드 3일 + 프롬프트 튜닝 + 평가)

기존 코드 재활용 비율: 약 90%. loader, chunker, embedder, retriever는 그대로.

### 데이터 준비

- 문서: 기존 `data_list.csv` 그대로 사용
- 평가: 기존 `eval_questions.csv` 중 50~100건만 뽑아서 사용
- 추가 데이터 불필요

## L — 입찰 비교 분석기

### 시나리오

RFP 2~3건을 올리면, 각 문서의 핵심 정보를 표로 비교하고 인사이트를 제시한다. "이 3개 입찰 중에 어느 게 제일 나은가"를 판단할 때 쓴다.

```
입력: RFP 3건 (A사업, B사업, C사업)

출력:
┌──────────┬──────────┬──────────┬──────────┐
│          │ A사업    │ B사업    │ C사업    │
├──────────┼──────────┼──────────┼──────────┤
│ 예산     │ 5억      │ 3.2억    │ 8억      │
│ 기간     │ 6개월    │ 4개월    │ 12개월   │
│ 자격     │ SW사업자  │ SW+보안   │ SW사업자  │
│ 서류     │ 8종      │ 12종     │ 6종      │
│ 독소조항 │ 지체 0.3% │ 없음     │ 해지일방  │
└──────────┴──────────┴──────────┴──────────┘

💡 C사업이 예산은 가장 크지만 안정적. A사업은 차순위.
   B사업은 예산이 작은데 서류 부담이 가장 큼 → 비추천.
```

### 필요한 Tool

| Tool | 하는 일 | 비고 |
|------|---------|------|
| `extract_info` (×N) | 각 RFP별 정보 추출 | M에서 재사용 |
| `normalize_fields` | RFP마다 다른 용어 통일 ("수행기간"="사업기간") | 신규 |
| `compare_budget` | 예산 크기 + 업계 평균 대비 분석 | 신규 |
| `compare_requirements` | 자격/서류 난이도 비교 | 신규 |
| `compare_risk` | 독소조항 유무 교차 비교 | 신규 |
| `recommend` | 종합 판단 → 가장 유리한 입찰 제안 | 신규 |
| `format_comparison` | 마크다운 비교표 생성 | M의 format_report 응용 |

### 수정 범위 + 소요 기간

```
수정: src/rag/pipeline.py                 2~3일
      → 복수 디렉터리 ingest 지원 (지금은 단일 디렉터리만 가능)
      → ingest 시 chunk에 "이 청크가 어느 RFP에서 왔는지" 메타데이터 추가

수정: src/rag/engines/langchain.py        1일
      → retriever: source_rfp 메타데이터로 필터링 지원
      → answerer: 비교 프롬프트 템플릿 추가

추가: src/rag/comparison.py               2일
      → normalize_fields, compare_entries, rank_entries 함수

추가: configs/experiments/rag/rag_comparison.yaml   반나절
```

**총 소요: 3~4주** (코드 1주 + 멀티 문서 검색 품질 튜닝 + 비교 평가셋 생성)

기존 코드 재활용 비율: 약 70%. M에서 StructuredOutput이 선행되어 있다는 가정.

### 데이터 준비

- 문서: 기존 100건 그대로 사용. config에 복수 raw_docs_dir 지정
- 평가셋: **새로 필요**. 비교 유형 질문은 현재 골든셋에 없음

  DE 작업: "A사업과 B사업의 예산 차이는?", "세 건 중 자격이 가장 까다로운 것은?" 같은 비교 질문을 30~50건 생성.

  ```csv
  question,expected_answer,expected_chunk_ids
  "A사업과 B사업의 예산 차이는?","A사업이 1.8억 더 많음",
  "세 건 중 자격이 가장 까다로운 것은?","B사업, 보안인증 추가 필요",
  ```

  기존 897건 중에서도 카테고리 다른 문서 2~3건씩 조합해서 비교 질문을 만들 수 있다. DE가 하루~이틀이면 생성 가능.

### L+ — RFP 추천 (우리 회사 맞춤)

#### 시나리오

여러 RFP 중 "우리 회사"에 가장 잘 맞는 입찰을 추천해준다. 단순 비교가 아니라 회사 프로필(자격증, 실적, 예산 밴드)과 매칭해서 점수를 매긴다.

```
입력: RFP 10건 + 우리 회사 프로필

출력:
  🥇 1순위: A사업 (적합도 92%)
     예산 5억, 6개월, 자격 충족 ✅, 실적 보유 ✅
  🥈 2순위: D사업 (적합도 78%)
     예산 3.2억, 3개월, 서류 6종 → 부담 적음
  ❌ 제외: C사업
     보안인증 필수 → 우리 회사 미보유
  ❌ 제외: F사업
     예산 1.2억 → 우리 최소 밴드(2억) 미달
```

#### 필요한 Tool

| Tool | 하는 일 |
|------|---------|
| `load_company_profile` | 우리 회사 자격증, 실적, 인력, 예산 밴드 로드 |
| `extract_info` (×N) | L과 동일, 각 RFP 요약 |
| `match_qualifications` | RFP 요구 자격 vs 회사 보유 자격 대조 |
| `score_fit` | 예산 밴드, 일정, 서류량 감안 종합 점수 산정 |
| `rank_recommend` | 점수 순 정렬 + 제외 사유 명시 |
| `format_recommendation` | 추천 리포트 + 적합도 시각화 |

#### 수정 범위 + 소요 기간

L 완료 후 추가 작업:

```
추가: src/rag/matcher.py                   1~2일
      → 회사 프로필 JSON 읽기 → RFP 요구사항과 매칭 → 점수화

추가: configs/experiments/rag/rag_recommend.yaml   반나절
      → company_profile_path: data/company_profile.json
```

**총 소요: L + 1주**. L 기반 위에 매칭 로직만 얹으면 된다.

#### 데이터 준비

- **회사 프로필**: JSON 한 파일로 끝. PM이 10분이면 작성 가능.

  ```json
  {
    "name": "우리회사",
    "qualifications": ["소프트웨어사업자", "정보보호관리체계(ISMS)"],
    "recent_projects": ["학사시스템 3억", "ERP 구축 7억"],
    "budget_band": {"min": "2억", "max": "10억"},
    "min_duration_months": 3,
    "max_duration_months": 24
  }
  ```

- **RFP 문서**: 기존 100건 그대로
- **평가셋**: 회사 프로필 2~3종 바꿔가며 추천 결과가 달라지는지 검증

## XL — RFP 컨설턴트 에이전트

### 시나리오

RFP 하나를 던지면, 정보 추출을 넘어서 **판단과 전략**을 제시한다. 실제 RFP 컨설턴트가 하는 일을 자동화한다고 보면 된다.

```
입력: RFP 1건

출력:
  📋 기본 정보
  사업명: OOO, 예산: 5억, 기간: 6개월, 발주: 조달청

  ⚠ 독소조항
  🔴 지체상금 0.3% (업계 평균 0.1~0.15%보다 높음)
  🟡 "계약해지 시 비용 수급자 부담" 조항 존재
  🟢 하도급 조항은 표준적

  📊 리스크 평가
  - 일정: 6개월 내 가능 ✅
  - 예산: 유사 규모 대비 10% 낮음 → 저가 리스크
  - 자격: 우리 회사 모두 충족 ✅

  🏢 경쟁 환경
  - 과거 유사 건 낙찰가: 평균 4.7억 (4.2~5.1억)
  - 경쟁사 A, B 주로 참여 → 예상 경쟁률 3:1

  🎯 전략 제안
  ✔ 참여 권장
  - 제안가: 4.8~5.0억
  - 차별화: 보안 인증 + 유사 실적 강조
  - 주의: 지체상금 협상 시도할 것
```

### 필요한 Tool (6단계, 16개)

**Phase 1 — 팩트 추출 (문서에 뭐가 있나)**

| Tool | 하는 일 |
|------|---------|
| `extract_facts` | 예산, 일정, 자격, 서류, 발주기관, 평가기준 추출 |
| `parse_bid_conditions` | 입찰방식, 공동도급 가능 여부 파악 |

**Phase 2 — 독소조항 탐지 (어디가 위험한가)**

| Tool | 하는 일 |
|------|---------|
| `scan_unfair_clauses` | "지체상금 과도", "계약해지 일방" 등 불리한 조항 탐지 |
| `scan_liability_shift` | "하도급 시 모든 책임은 수급자에게" 같은 책임 전가 찾기 |
| `rate_clause_severity` | 발견된 조항마다 위험도 1~5점 산정 |

**Phase 3 — 리스크 진단 (얼마나 위험한가)**

| Tool | 하는 일 |
|------|---------|
| `check_schedule_viability` | 기간 대비 서류량·인력 감안 현실성 점검 |
| `check_budget_sufficiency` | 예산 대비 요구 규모가 적정한지 판단 |
| `check_qualification_gap` | 우리 회사 보유 자격 vs 요구 자격 대조 |

**Phase 4 — 경쟁 분석 (시장에서 어떤 위치인가)**

| Tool | 하는 일 |
|------|---------|
| `search_similar_bids` | 과거 유사 입찰의 낙찰가·참여사 검색 |
| `analyze_competitor_pattern` | 경쟁사가 자주 참여하는 유형인지 |
| `estimate_winning_bid` | 예상 낙찰가 밴드 추정 |

**Phase 5 — 전략 판단 (그래서 어떻게 할 것인가)**

| Tool | 하는 일 |
|------|---------|
| `decide_participation` | 참여 O/X + 근거 |
| `suggest_bid_price` | 제안가 권장 범위 |
| `draft_proposal_checklist` | 제안서 목차 + 주의사항 초안 |

**Phase 6 — 리포트 종합**

| Tool | 하는 일 |
|------|---------|
| `compile_report` | Phase 1~5 결과를 한 장짜리 문서로 |

### 수정 범위 + 소요 기간

**공통 선행 작업 (M에서 해결됨)**:
```
answerer StructuredOutput 지원   → M 단계에서 완료
```

**기존 코드 수정**:
```
수정: src/rag/engines/langchain.py       2일
      → 멀티 문서 retriever (L에서 선행)
      → 컨설턴트 페르소나 프롬프트 추가

수정: src/rag/pipeline.py                 3~4일
      → 단일 체인 → 6 Phase 상태 머신으로 재작성
      → Phase 간 데이터 누적 및 실패 시 재시도 로직

변경: src/rag/evaluation/                 3~4일
      → "정답 매칭" → "LLM-as-judge 품질 평가"로 전면 교체
```

**신규 모듈**:
```
신규: src/agent/loop.py                   3일
      → 6 Phase 순차 실행 + 결과 누적 + 실패 분기

신규: src/agent/tools.py                  2일
      → 16개 Tool 정의 + 레지스트리

신규: src/agent/prompts.py                1일
      → 컨설턴트 페르소나 시스템 프롬프트

신규: src/agent/rules.py                  2일
      → 독소조항 패턴 매칭 (Rule 엔진)

신규: configs/agent/rag_consultant.yaml    반나절
```

**추가 인프라**:
| 요소 | 용도 | 소요 |
|------|------|------|
| 나라장터 낙찰 DB | Phase 4 경쟁 분석 | API 연동 1주 |
| 피드백 루프 | 사람이 분석 오류 수정 | 지속 |
| 감사 로그 | 판단 근거 추적 | 1일 |

**총 소요: 2개월+** (M/L 선행 완료 가정 시)

기존 코드 재활용 비율: 약 50%. loader, chunker, embedder는 그대로. answerer와 retriever는 일부 수정. pipeline은 대폭 수정. evaluation은 완전 교체.

### 데이터 준비

- **문서**: 기존 100건 그대로 사용
- **Rule 패턴**: 독소조항 정규표현식 패턴 — PM이 10~20건 정도 실제 RFP에서 샘플링해서 정의
- **낙찰 데이터**: 나라장터 과거 낙찰 공고 — 공공데이터포털 API로 수집. 약 1,000~5,000건
- **평가셋**: **완전히 새로 필요**

  컨설팅 품질을 평가하는 건 정답이 하나가 아니므로:

  1. DE가 30~50건의 "컨설팅 질문" 생성
  2. 각 질문에 대해 LLM-as-judge(별도 평가용 LLM)가 응답 품질 1~5점 채점
  3. 사람 리뷰: 컨설턴트 경험자에게 10~20건 샘플 평가 요청

  평가 체계를 만드는 것 자체가 1~2주 소요되는 작업이다.

## 역할 분담

각 서비스 레벨에서 누가 무슨 일을 해야 하는가.

### M — RFP 자동 요약 리포트

| 담당 | 할 일 |
|------|-------|
| **EL** | answerer에 StructuredOutput 기능 추가 (Pydantic 스키마 → LLM 출력 강제). 요약 전용 파이프라인 `run_rag_summarize` 구현. 요약 프롬프트 튜닝 + 실험 + 평가. |
| **DE** | 추가 데이터 없음. 기존 평가셋 897건 중 50~100건만 선별해서 M 전용 평가셋으로 재구성. "예산+일정+자격 한꺼번에 추출" 복합 질문에 맞게 expected_answer 포맷 조정. |
| **PM** | 요약 항목 정의: "이 RFP 요약에 어떤 항목이 들어가야 쓸모 있는가?" 결정. 프롬프트 튜닝 시 정성 피드백. EL의 평가 결과 보고 다음 단계 진행 결정. |

### L — 입찰 비교 분석기

| 담당 | 할 일 |
|------|-------|
| **EL** | 멀티 문서 ingest 지원 (복수 raw_docs_dir + chunk에 source_rfp 메타데이터 추가). retriever 필터링. 비교 파이프라인 구현 (`src/rag/comparison.py`). 비교 프롬프트 템플릿 제작 + 튜닝. |
| **DE** | 비교 평가셋 **30~50건 새로 생성**. 예: "A사업과 B사업의 예산 차이는?", "세 건 중 자격이 가장 까다로운 것은?". 문서 2~3건 조합하는 쿼리 설계. |
| **PM** | 비교 항목 정의: "어떤 기준으로 입찰을 비교해야 PM에게 의미 있는가?" 결정. 용어 정규화 대상 목록 확정. 비교 인사이트 포맷 결정 ("추천" 로직의 기준). |

### L+ — RFP 추천 (우리 회사 맞춤)

| 담당 | 할 일 |
|------|-------|
| **EL** | L 완료 후 `src/rag/matcher.py` 추가. JSON 프로필 로드 + RFP 요구사항 매칭 + 점수화. 추천 포맷 프롬프트 튜닝. |
| **DE** | 회사 프로필 JSON 템플릿 작성. L+ 평가셋 생성 (프로필 변경 시 추천 결과 검증). |
| **PM** | 회사 프로필 초안 작성 (자격증, 실적, 예산 밴드). 매칭 기준 정의 (어디까지 Pass, 어디까지 Fail). |

### XL — RFP 컨설턴트 에이전트

| 담당 | 할 일 |
|------|-------|
| **EL** | `src/agent/` 모듈 전체 신규 개발. 6 Phase 상태 머신. 16개 Tool 구현 + 레지스트리. Rule 엔진. LLM-as-judge 평가 체계 전면 재설계. pipeline을 Phase 기반으로 재작성. |
| **DE** | 컨설팅 평가셋 30~50건 생성. 나라장터 낙찰 DB 수집·정제 (Phase 4 데이터). Rule 패턴 정의 시 실제 RFP 샘플링 지원. |
| **PM** | 컨설팅 워크플로우 6 Phase 설계. 독소조항 판단 기준 정의 ("어디까지가 위험인가"). 경쟁 분석 시 참고할 업계 지식 제공. 최종 리포트 품질 리뷰 (사람 평가). LLM-as-judge 평가 항목 설계. |

### 역할별 부담 비교

| | M | L | L+ | XL |
|---|----|----|-----|-----|
| **EL 부담** | 낮음 (모듈 1개 수정) | 중간 (모듈 2~3개) | 중간 (L 위에 matcher 추가) | 높음 (신규 모듈 4개 + 평가 체계 재설계) |
| **DE 부담** | 낮음 (기존 평가셋 선별) | 중간 (비교 평가셋 30~50건) | 낮음 (프로필 JSON + 검증) | 높음 (평가셋 + 낙찰DB + 패턴) |
| **PM 부담** | 낮음 (항목 정의) | 중간 (비교 기준 정립) | 낮음 (프로필 작성 + 매칭 기준) | 높음 (워크플로우 설계주도 + 품질 리뷰) |

## 추천 방향

**M을 최우선 후보로 제안한다.** 이유:

1. PM 실무에 바로 도움됨. "문서 요약"은 누구나 공감하는 니즈.
2. 팀 부담이 가장 작다. 코드 변경량이 적고, 평가도 기존 방식 유지 가능.
3. StructuredOutput은 M에서 만들면 L과 XL에서도 그대로 쓸 수 있다. M은 기술적 선행 투자.
4. M이 성공하면 L로 자연스럽게 확장 가능. 실패해도 손실이 작다.

**일정**:
- M 성공 (1~2주) → L 착수 (3~4주) → L+ 추천 (1주) → L 성공 시 XL 논의
- 각 단계에서 평가 결과 보고 전체 회의로 다음 단계 진입 여부 결정
