# 프로젝트 일정 가이드

이 문서는 팀이 공식 일정과 작업 우선순위를 같은 기준으로 보기 위한 일정 요약입니다.

킥오프에서 프로젝트 목표를 공유한 뒤, 두 번째로 이 문서를 열어 발표일까지 남은 평일과 단계별 집중 과제를 맞춥니다.

## 전체 기간

| 구분 | 날짜 | 설명 |
| --- | --- | --- |
| 프로젝트 설명 | 2026-06-17 | 강사님께 프로젝트 설명을 듣고, 팀 역할과 운영 방식을 맞춥니다. |
| 실질 진행 | 2026-06-18 ~ 2026-07-07 | 평일 기준으로 데이터, 실험, 문서, 발표 자료를 준비합니다. |
| 발표 | 2026-07-08 | 결과, 한계, 개선 방향을 발표합니다. |

공식 일정은 평일 기준으로 관리합니다. 주말은 기본 계획에 넣지 않고, 필요한 경우에만 별도로 논의합니다.

## 진행 기준

| 기간 | 집중할 일 | 완료 기준 |
| --- | --- | --- |
| 2026-06-18 ~ 2026-06-19 | 환경, 문서, 역할, 보드 정리 | 팀원이 실행 방법과 Issue/PR 흐름을 이해합니다. |
| 2026-06-22 ~ 2026-06-26 | 데이터 후보와 RAG baseline 실행 | 문서 loader, 평가 질문, 기본 RAG 산출물이 확인됩니다. |
| 2026-06-29 ~ 2026-07-03 | retriever, answerer, config 실험 | 비교 가능한 config와 metric, 실패 사례가 남습니다. |
| 2026-07-06 ~ 2026-07-07 | 발표 자료와 데모 흐름 정리 | 질문, 답변, citation, 한계가 발표 흐름으로 연결됩니다. |
| 2026-07-08 | 발표 | 팀이 설명할 결과와 다음 개선 방향이 정리되어 있습니다. |

## 일정 운영 원칙

- 모든 작업은 Issue나 Kanban 카드로 남깁니다.
- 일정표는 목표를 맞추기 위한 기준이고, 세부 작업은 보드에서 조정합니다.
- RAG 품질은 답변만 보지 않고 retrieval 결과, citation, failure artifact를 함께 봅니다.
- 발표 전에는 성공 사례뿐 아니라 실패 사례와 한계도 같이 정리합니다.
