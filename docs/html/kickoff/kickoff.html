프로젝트 킥오프 가이드

개발 경험이 많지 않아도 전체 흐름을 이해할 수 있도록 쉽게 설명하되, 실제 작업에 필요한 기준과 산출물도 함께 정리한 문서입니다.

마지막 통합 줄이기 데이터 약속표 실험 기록 GitHub 작업 게시판

1. 우리가 먼저 맞추려는 것

팀 프로젝트에서 가장 힘든 순간은 보통 마지막입니다. 각자 열심히 만들었는데 데이터 형식, 모델 입력, 결과 파일 위치가 달라서 합치기 어려워질 수 있습니다.

이번에는 그런 일을 줄이기 위해 처음부터 몇 가지 약속을 맞춥니다.

데이터어떤 형태로 정리해서 넘길지 맞춥니다.
결과물각자 무엇을 남길지 미리 정합니다.
공유작업과 막힌 점을 GitHub에 남깁니다.
한 문장으로 말하면, 각자 만든 작업물이 마지막에 자연스럽게 붙도록 미리 약속을 정하는 것입니다.

2. 전체 작업 흐름

데이터 준비
-> 데이터 형식 확인
-> 모델 학습
-> 결과 저장
-> 데모 앱 연결
-> 보고서/발표 정리

중요한 것은 누가 어느 단계의 결과물을 다음 사람에게 넘기는지입니다.

Data 담당 -> Experiment 담당: 정리된 데이터와 데이터 설명
Experiment 담당 -> App 담당: 모델 파일과 입력/출력 형식
Experiment 담당 -> Report 담당: 실험 결과와 최종 모델 선택 이유
App 담당 -> Report 담당: 데모 화면과 사용 흐름
PM -> 전체: 일정, 기준, 통합 상태

3. 역할은 담당 구역입니다

역할은 일을 제한하기 위한 것이 아닙니다. 누가 어떤 부분을 책임지고 정리할지 정하는 기준입니다. 필요하면 서로 도와도 됩니다.

역할쉽게 말하면맡는 일반드시 남길 것
PM전체 흐름을 정리하고 연결하는 사람일정, 기준, 역할 조율, 통합 관리운영 기준, 일정, 결정사항, 통합 체크
Data Engineer데이터를 모델이 쓸 수 있게 정리하는 사람데이터 정리, 데이터 품질 확인, 데이터 설명train/valid/test, class_map, dataset_info, 데이터 이슈
Experiment Lead모델을 고르고 실험하는 사람모델 선택, 학습 실행, 결과 비교config, metrics, 모델 파일, 실험 해석
Application Engineer결과를 사용자가 볼 수 있게 만드는 사람데모 화면, API, 입력/출력 연결실행 가능한 데모, 사용 방법, 화면 캡처
Presentation Lead프로젝트를 설명 가능한 이야기로 만드는 사람보고서, 발표자료, 발표 흐름 정리최종 보고서, 발표자료, 발표 흐름, 리허설 체크
각 담당자가 자기 파트의 설명과 근거를 남기고, 발표/보고서 담당자는 그 자료를 모아 하나의 흐름으로 정리합니다.
A안: 웹앱/데모 시도
PM / Data Engineer / Experiment Lead / Application Engineer / Presentation Lead

B안: 모델/실험 강화
PM / Data Engineer / Model Engineer / Experiment Lead / Presentation Lead

Application Engineer를 맡고 싶은 팀원이 있으면 A안으로 갑니다. 웹앱 담당이 없거나 일정이 촉박하면 B안으로 가고, 데모는 Colab Notebook과 결과 시각화 중심으로 낮춥니다.

4. 데이터 약속표

우리는 데이터를 아무 형태로나 넘기지 않고, 모두가 이해할 수 있는 공통 형태로 맞춥니다. 이것을 Data Contract, 쉽게 말해 데이터 약속표라고 부릅니다.

# 텍스트 분류 예시
text,label
이 영화는 재미있다,positive
이 앱은 자주 멈춘다,negative

# 이미지 분류 예시
image_path,label
images/img_000001.jpg,cat

데이터 담당자가 준비할 것:

  • train.csv: 학습용 데이터
  • valid.csv: 학습 중 성능 확인용 데이터
  • test.csv: 최종 확인용 데이터
  • class_map.json: label 이름과 번호의 대응표
  • dataset_info.json: 데이터 버전, 클래스 수, split 개수, 주의사항

이 확인을 자동으로 해주는 것이 validate_data.py입니다.

5. 전처리와 증강은 구분합니다

구분목적예시적용 위치
데이터 정리데이터가 파이프라인에 들어가게 만들기label 정리, split 생성, 결측 확인데이터 준비 단계
입력 변환모델이 받을 수 있는 형태로 바꾸기tokenization, resize, normalize학습/예측 코드
학습용 증강성능을 높이기 위한 변형random crop, noise, synonym replace보통 train에만
공통 데이터는 너무 많이 바꾸지 않습니다. 학습 성능을 위한 증강은 실험 설정으로 관리하고, 검증/테스트/실제 예측에는 랜덤 증강을 적용하지 않습니다.

6. 실험 기록은 왜 남기나

모델 실험을 하다 보면 “어떤 설정으로 돌렸더라?”를 잊기 쉽습니다. 그래서 실험마다 설정, 결과, 모델 파일 위치, 실패 이유를 남깁니다.

다시 실행잘 된 실험을 재현할 수 있습니다.
실패 방지같은 실패를 반복하지 않을 수 있습니다.
보고서 근거발표와 보고서에 넣을 근거가 생깁니다.
config.yaml
metrics.json
history.csv
run_info.json
best_model
README.md

7. GitHub는 작업 게시판입니다

GitHub는 코드 저장소이기도 하지만, 이 프로젝트에서는 작업 게시판처럼 사용합니다.

Issue에서 할 일 확인
-> 자기 작업 진행
-> PR로 변경 내용 공유
-> 팀원이 확인
-> 통합

PR에는 작업 내용, 변경 이유, 실행해서 확인한 것, 관련 산출물 경로, 리뷰가 필요한 부분을 적습니다.

8. 우리가 쓸 폴더를 쉽게 보면

data/       데이터
configs/    실험 설정
src/        실제 코드
app/        데모 앱
docs/md/    원본/관리용 Markdown 문서
docs/html/  공유/설명용 HTML 문서
reports/    실험 기록과 보고서 자료
tests/      자동 확인 코드

실험을 실행하면 experiments/ 폴더에 실험별 결과가 한 덩어리로 저장됩니다.

10. Daily Report는 짧게 씁니다

Daily Report는 긴 보고서가 아닙니다. 오늘 한 일, 관련 링크, 막힌 점만 남기면 됩니다.

오늘 한 일:
- text_processed 데이터 초안 작성
- validate_data.py 실행 확인

관련 링크:
- Issue #3
- PR #5

막힌 점:
- label 기준을 positive/negative로 갈지, 0/1로 갈지 확인 필요
막힌 점은 숨기는 것이 아니라 빨리 공유하는 것이 좋습니다. 그래야 팀 전체 속도가 빨라집니다.
