프로젝트 파이프라인 쉬운 설명

이 문서는 코드를 처음 보는 팀원도 “왜 이런 구조를 만들었고, 실험 하나가 어떤 순서로 돌아가며, 결과가 어디에 남는지” 이해할 수 있도록 만든 설명 자료입니다.

실험 흐름 Config 중심 구조 RAG 이해 산출물과 재현성

먼저 큰 그림

Config 실험 주문서입니다. 어떤 데이터와 모델을 쓰고, 결과를 어디에 저장할지 적습니다.
Scripts 실행 버튼입니다. 팀원은 복잡한 내부 코드를 몰라도 같은 명령으로 실행합니다.
src 실제 작업자입니다. 데이터 검증, 학습, 검색, 답변, 평가를 수행합니다.
Experiments 실험 영수증입니다. 설정, 결과, 로그, 실패 원인을 나중에 다시 확인할 수 있게 남깁니다.

실험 하나가 지나가는 길

핵심은 실험의 기준이 코드가 아니라 config라는 점입니다. 코드를 매번 고치는 대신 config를 바꿔서 조건을 비교하면, 팀원끼리 결과를 훨씬 쉽게 재현할 수 있습니다.

왜 이렇게 나눴나

나누지 않으면 생기는 문제

사람마다 실행 방식이 달라지고, 결과 파일 위치가 흩어지고, 어떤 데이터와 모델로 나온 결과인지 나중에 찾기 어려워집니다.

나누면 좋아지는 점

같은 config와 같은 script로 실행하므로 재현이 쉬워지고, 실험 결과가 일정한 폴더 구조로 남아 발표와 회고에 쓰기 좋아집니다.

Config는 무엇을 정하나

사람 말로 풀면

“이번 실험은 이 데이터를 쓰고, 이 모델을 쓰고, 결과는 이 폴더에 저장하고, 실패하면 로그를 남겨줘”라고 적는 주문서입니다.

파일로 보면

experiment:
  name: rag_smoke_test

paths:
  output_dir: experiments/rag_smoke_test

rag:
  retriever:
    method: hybrid
    top_k: 3

분류 모델과 RAG의 차이

구분 분류 모델 RAG
목표 입력 하나를 보고 정해진 label 중 하나를 예측합니다. 문서에서 근거를 찾아 질문에 답하고 출처를 남깁니다.
예시 문장이 긍정인지 부정인지 분류합니다. RFP 문서에서 예산, 자격 요건, 일정 등을 찾아 답합니다.
중요한 산출물 metric, best model, prediction chunks, retrieved chunks, answer, citations, 실패 분석
평가 포인트 accuracy 같은 예측 성능 관련 근거를 찾았는지, 답변에 근거가 있는지

RAG는 무엇을 하는가

문서를 읽습니다

PDF, HWP, DOCX, TXT 같은 원본 문서를 읽고 표준 형태의 document row로 바꿉니다.

작은 근거 단위로 나눕니다

긴 문서를 한 번에 다루기 어렵기 때문에 검색 가능한 chunk로 나눕니다.

검색 가능한 형태로 바꿉니다

각 chunk를 embedding으로 바꾸거나 keyword 검색에 쓸 수 있게 준비합니다.

질문과 관련 있는 근거를 찾습니다

질문이 들어오면 keyword, semantic, hybrid 방식으로 관련 chunk를 고릅니다.

근거를 바탕으로 답합니다

답변과 함께 어떤 chunk를 근거로 삼았는지 citation을 남깁니다.

실험 결과 폴더를 보면 알 수 있는 것

성공했을 때

metrics.json, history.csv, config.yaml, run_info.json을 보면 어떤 조건으로 어떤 결과가 나왔는지 확인할 수 있습니다.

실패했을 때

run_status.json와 failure.log를 보면 어느 단계에서 왜 실패했는지 추적할 수 있습니다.

experiments/rag_smoke_test/
|-- config.yaml
|-- metrics.json
|-- run_status.json
|-- parsed_documents.csv
|-- chunks.csv
|-- embeddings.jsonl
|-- answers.jsonl
`-- failure.log

현재 구현된 것과 나중에 붙일 것

영역 현재 가능한 것 나중에 붙일 수 있는 것
분류 모델 이미지 smoke, 텍스트 keyword, HuggingFace tiny/fine-tuning 흐름 실제 프로젝트 데이터에 맞춘 base model 교체
RAG 문서 처리 txt, pdf, docx, hwpx, hwp 로딩과 chunking 문서 품질에 맞춘 전처리와 section 추출 고도화
RAG 검색 keyword, semantic, hybrid 비교 FAISS, Chroma, Elasticsearch, reranker
RAG 답변 local extractive answer OpenAI 또는 HuggingFace LLM answer adapter
운영 기능 checkpoint, resume, scheduler, early stopping, backup 정책 대용량 문서용 세밀한 offset resume

팀원에게 설명할 때의 핵심 문장

이 저장소는 모델 하나를 정답처럼 고정하는 저장소가 아니라, 여러 실험을 같은 방식으로 실행하고 비교하기 위한 기본 장치입니다.

팀원은 config를 바꾸고 script를 실행하면 되고, 결과는 experiments에 자동으로 남습니다. RAG 프로젝트가 되면 문서에서 근거를 찾고 답변과 citation을 남기는 흐름으로 확장됩니다.
