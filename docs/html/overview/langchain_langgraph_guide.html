LangChain & LangGraph 개념 정리

이론 → 코드 예제 → 우리 RAG 프로젝트 매핑까지 한 번에

0. LangChain이 왜 필요한가

LLM 혼자서는 못 하는 것

한계설명예시
환각 (Hallucination) 학습 데이터에 없는 정보를 물어보면 그럴듯하게 지어냄. 특히 최신 정보나 기업 내부 문서에 취약 "우리 회사 RFP 마감일이 언제야?" → 모르는데도 날짜를 만들어냄
외부 세계 단절 웹 검색, DB 조회, 파일 읽기, API 호출 등 스스로 실행할 수 없음. 오직 텍스트 생성만 가능 날씨 API를 호출하거나 PDF 파일을 직접 읽지 못함
컨텍스트 윈도우 제한 한 번에 처리할 수 있는 토큰 수에 상한선이 있음. 긴 문서를 통째로 넣을 수 없음 200페이지 RFP 문서를 한 번에 입력 불가

→ LangChain은 이 3가지 한계를 극복하기 위해 등장한 오픈소스 라이브러리. LLM을 중심에 두고 그 주변에 문서 로더, 검색기, 벡터DB, 도구, 메모리를 연결해주는 접착제 역할.

1. 채팅 모델 로드 — 하나의 함수로 모든 LLM을

이론: Provider 추상화

OpenAI, Google, Anthropic, Ollama, HuggingFace 등 LLM 제공자는 각자 API 방식이 다름. LangChain은 init_chat_model()이라는 단일 진입점으로 이 차이를 감춤. "모델 이름"과 "제공자 이름"만 바꾸면, 나머지 코드는 한 줄도 고칠 필요 없이 다른 LLM으로 교체 가능. 

# ===== 기본 로드 =====
from langchain.chat_models import init_chat_model

model = init_chat_model(
    "gemini-2.5-flash-lite",    # 모델 이름 (제공자마다 다름)
    model_provider="google-genai",      # 제공자 식별자
)

# ===== 옵션 포함 로드 =====
model = init_chat_model(
    "gemma4:e2b",                   # Ollama 로컬 모델
    model_provider="ollama",            # 제공자: ollama
    temperature=0.5,                  # 0.0 = 정직/일관적, 1.0 = 창의적/다양함
    timeout=600,                       # 응답 대기 최대 시간(초)
    max_tokens=25000,                  # 생성할 최대 토큰 수
    num_ctx=512,                      # (Ollama 전용) 컨텍스트 윈도우 크기
)

# ===== 사용 =====
result = model.invoke("랭체인이 뭐야?")  # → AIMessage 객체로 응답
print(result.content)                     # → 텍스트만 추출

Googlegoogle-genailangchain-google-genai무료 할당량 있음한국어 성능 편차
Ollamaollamalangchain-ollama로컬 무료, 오프라인 가능무거운 모델은 GPU 필요

→ 우리 프로젝트 매핑: src/rag/engines/langchain.py:120-131 — _build_chat_model() 메서드가 Config의 rag.answerer.provider 값을 읽어서 ollama면 ChatOllama, openai면 ChatOpenAI로 분기. 팀원이 Config만 바꾸면 LLM 교체 가능.

파이프라인에서의 위치: 이 모델은 run_rag_chat() → engine.answer() → _build_chat_model() 경로로 호출되어, 검색된 문서 조각(context)과 함께 최종 답변을 생성함.

2. Message — 대화를 역할별로 구조화

이론: 왜 Message가 필요한가

LLM에 그냥 문자열을 던질 수도 있지만, 그러면 "누가 한 말인지" 구분이 안 됨. Message는 발화자의 역할(Role)을 태그처럼 붙여서 LLM이 맥락을 더 정확히 이해하게 함. 특히 SystemMessage는 LLM에게 "너는 이런 역할이야"라고 행동 지침을 주는 데 필수적.

4가지 Message 타입

타입누가 보내는지용도실제 대화 예시
SystemMessage 개발자 모델의 페르소나/행동 규칙 설정. 대화 내내 유지됨 "너는 RFP 문서 전문 분석가야. 근거에 없는 내용은 추측하지 마."
HumanMessage 사용자 질문, 요청, 입력. 멀티모달이면 이미지/음성도 포함 가능 "입찰 마감일이 언제야?"
AIMessage LLM 생성된 답변. 내부에 tool_calls 배열이 있으면 "도구를 호출하겠다"는 의사표시 "마감일은 2026년 7월 31일입니다."
ToolMessage 도구 실행 결과 함수가 반환한 값을 LLM에게 다시 전달할 때 사용 "get_tomorrow_date" → "2026-06-16" 
# ===== 기본 사용: SystemMessage + HumanMessage =====
from langchain.messages import SystemMessage, HumanMessage, AIMessage

messages = [
    SystemMessage("너는 식물 전문가다. 식물 관련 질문에만 답하라."),
    HumanMessage("마지나타 키우는 기본 환경 알려줘"),
]
result = model.invoke(messages)

# ===== 대화 이어가기: 이전 응답을 messages에 추가 =====
messages.append(result)                       # AIMessage 추가
messages.append(HumanMessage("그중 가장 중요한 건?"))
result2 = model.invoke(messages)              # 이전 맥락을 기억하고 답변

# ===== stream()으로 한 토큰씩 출력 =====
for chunk in model.stream(messages):
    print(chunk.content, end="", flush=True)  # 사용자 경험 개선

# ===== pretty_print()로 가독성 있게 =====
result2.pretty_print()  # ================================ Ai Message ================================
→ 우리 프로젝트 매핑:

① answerer.py (로컬 추출형 답변): src/rag/answerer.py:9-38 — build_answer() 함수. 검색된 chunk 중 질문과 가장 토큰이 많이 겹치는 문장을 골라서 답변으로 반환. LLM을 호출하지 않고 순수 규칙 기반으로 동작 → 비용 0, 빠름.

② langchain.py (LLM 답변): src/rag/engines/langchain.py:326-335 — _build_prompt() 함수. Config의 rag.answerer.provider가 ollama나 openai인 경우에만 호출됨. SystemMessage + Context + 질문을 하나의 문자열로 조합 → LLM에 전달.

③ 파이프라인에서의 위치: src/rag/pipeline.py:162-164 — run_rag_chat() → engine.answer() 검색 결과와 질문을 answerer에 넘겨서 최종 답변 + citation을 생성.

3. Tool — LLM에게 "이런 기능 있어" 알려주기

이론: 도구(Tool)란

LLM은 텍스트 생성 외에는 아무것도 못 함. 날짜 계산, 웹 검색, DB 조회는 모두 외부 함수가 필요. Tool은 "이런 함수가 있으니 필요하면 호출해도 돼"라고 LLM에게 함수의 존재를 알려주는 메커니즘. LLM은 각 Tool의 docstring(설명서)을 읽고 적절한 도구를 선택해서 호출 여부를 결정함. 

from langchain.tools import tool
from datetime import datetime, timedelta

# ===== Tool 정의 규칙 =====
# 1. @tool 데코레이터 필수
# 2. docstring(함수 설명) 필수 → LLM이 이걸 읽고 언제 쓸지 판단
# 3. 타입 힌트 필수 → LLM이 인자에 어떤 값을 넣을지 이해

@tool
def get_tomorrow_date() -> str:
    """
    내일의 날짜를 'YYYY-MM-DD' 형식으로 반환합니다.
    오늘 날짜가 필요할 때 사용하세요.
    """
    tm = datetime.now() + timedelta(days=1)
    return tm.strftime("%Y-%m-%d")


@tool
def get_weather(location: str, date: str) -> str:
    """
    입력된 날짜와 위치를 기반으로 날씨 정보를 반환합니다.
    먼저 get_tomorrow_date로 날짜를 확인한 후 호출하세요.

    Args:
        location: 도시 이름 (예: "서울", "부산")
        date: 조회할 날짜 (YYYY-MM-DD 형식)
    """
    return f"{location}의 날씨는 맑음"


# ===== 도구 수동 실행 확인 =====
tools = [get_tomorrow_date, get_weather]
print(tools[0].name)           # → "get_tomorrow_date"
print(tools[0].description)    # → docstring 내용 (LLM이 읽는 부분)
print(tools[0].invoke({}))      # → "2026-06-16" (직접 실행)
→ 우리 프로젝트 매핑:

LangChain의 Tool 개념을 우리 프로젝트에서는 직접 사용하지 않음. 대신 src/rag/retriever.py가 Tool과 동일한 역할을 함 — 질문을 입력받아 관련 문서를 검색해서 반환하는 함수. 다만 @tool 데코레이터 없이 순수 파이썬 함수로 구현되어 있고, LLM이 호출 여부를 판단하는 게 아니라 pipeline.py가 무조건 호출하는 구조임.

이 설계는 의도적인 선택 — RFP 문서 질문은 99% 검색이 필요하므로, LLM이 "검색 스킵"할 기회를 주지 않고 무조건 검색 → 답변 순서로 강제하는 게 더 안정적이기 때문.

4. bind_tools → create_agent (진화 과정)

이론: 왜 그냥 bind_tools로는 부족한가

model.bind_tools([...])는 LLM에게 "이 도구들을 쓸 수 있어"라고 알려줄 뿐, 호출 결과를 다시 LLM에 전달하는 반복 루프는 자동으로 해주지 않음. 개발자가 직접 "LLM 응답 → tool_calls 확인 → 도구 실행 → ToolMessage 생성 → 다시 LLM 호출"의 5단계를 while문으로 구현해야 함.

반면 create_agent()는 이 루프를 LangGraph가 내부 그래프로 자동 생성해줌. LLM이 "더 이상 쓸 도구가 없어"라고 판단할 때까지 ReAct 루프를 무한 반복.

수동 bind_tools 방식 (개발자가 반복문 직접 구현)

# ===== 1. 도구 바인딩 =====
model_with_tools = model.bind_tools([get_tomorrow_date, get_weather])

# ===== 2-6. 수동 ReAct 루프 =====
messages = [HumanMessage("내일 서울 날씨 어때?")]
response = model_with_tools.invoke(messages)  # ② LLM 호출
messages.append(response)

for tc in response.tool_calls:             # ③ tool_calls 확인
    if tc['name'] == 'get_tomorrow_date':
        result = get_tomorrow_date.invoke(tc) # ④ 도구 실행
    elif tc['name'] == 'get_weather':
        result = get_weather.invoke(tc)

    messages.append(result)        # ⑤ ToolMessage로 감싸서 추가

response2 = model_with_tools.invoke(messages) # ⑥ 다시 LLM 호출
# ... tool_calls가 더 없으면 최종 답변

자동 create_agent 방식 (LangGraph가 알아서 반복)

from langchain.agents import create_agent

agent = create_agent(
    model=model,
    tools=[get_tomorrow_date, get_weather],
    system_prompt="도구를 적절히 사용하여 답변하라",
)

# 단 한 줄로 수동 5단계가 전부 자동화됨
result = agent.invoke({
    "messages": [HumanMessage("내일 서울 날씨 어때?")]
})
create_agent 내부의 자동화된 ReAct 루프:
LLM 판단→ 도구 호출→ 결과 관찰→ LLM 재판단→ 최종 답변
→ 우리 프로젝트 매핑: 의도적 미사용

create_agent를 쓰지 않는 이유는 간단함: RFP 문서 기반 질문에서 검색을 스킵할 이유가 없음. LLM에게 검색 여부를 맡기면:
  • LLM이 검색을 건너뛰고 자신의 학습 지식으로 답변 → 환각 발생 가능
  • LLM 호출이 2회 이상으로 증가 → 속도 저하
  • 평가 결과가 비결정적 → 재현성 문제
대신 src/rag/pipeline.py:154-170 — run_rag_chat() 가 retrieve → answer 순서를 무조건 강제하는 선형 파이프라인으로 안정성 확보. 이건 1번 노트북의 Middleware Agent(강제 검색) 패턴과 정확히 같은 철학.

5. Memory — 대화를 기억하는 법

이론: stateless LLM에 기억력을 주입

LLM은 기본적으로 무상태(stateless) — 매 호출이 독립적이고 이전 대화를 기억하지 못함. Memory는 대화 히스토리를 외부 저장소에 보관했다가, 다음 호출 시 자동으로 주입해주는 구조. thread_id로 각 대화 세션을 구분하고, checkpointer가 저장/복원을 담당. 

from langchain.agents import create_agent
from langgraph.checkpoint.memory import InMemorySaver

agent = create_agent(
    model=model,
    tools=[...],
    checkpointer=InMemorySaver(),  # ← 메모리 추가
)

# 첫 번째 질문
result1 = agent.invoke(
    {"messages": [HumanMessage("내일 서울 날씨?")]},
    {"configurable": {"thread_id": "0001"}}  # 대화 식별자
)
# → "내일 서울 날씨는 맑습니다"

# 두 번째 질문 — 이전 대화 자동 기억!
result2 = agent.invoke(
    {"messages": [HumanMessage("그럼 우산 필요해?")]},
    {"configurable": {"thread_id": "0001"}}  # 같은 thread_id
)
# → "날씨가 맑으니 우산은 필요없습니다" (이전 '맑음'을 기억!)
Memory 종류저장 위치용도
InMemorySaverRAM (프로세스 종료 시 소멸)개발/테스트
SqliteSaverSQLite 파일 (영구)소규모 실서비스
PostgresSaverPostgreSQL DB대규모 서비스
→ 우리 프로젝트 매핑: 현재 불필요
질문 하나에 답변 하나를 생성하는 단발성 구조라서 Memory가 필요 없음. 나중에 대화형 RFP 챗봇으로 확장하면 InMemorySaver부터 붙이면 됨.

6. Middleware — LLM 호출 전/후에 끼어들기

이론: Tool은 LLM이 선택하지만, Middleware는 개발자가 강제

Tool은 LLM이 "지금 이 도구를 써야겠다"고 자율적으로 판단. Middleware는 LLM의 의사와 무관하게 특정 시점에 개발자가 원하는 로직을 강제로 실행. 주로 프롬프트 변경, 토큰 관리, 검증, 로깅 등에 사용.

데코레이터실행 시점주 용도
@before_modelLLM 호출 직전프롬프트 검증, PII 마스킹
@after_modelLLM 응답 직후응답 검증, 로깅
@dynamic_prompt시스템 프롬프트 수정RAG 검색 결과 주입
@wrap_tool_call도구 호출 전후도구 실행 시간 측정, 재시도

핵심 패턴: @dynamic_prompt로 강제 검색 RAG

from langchain.agents.middleware import dynamic_prompt, ModelRequest

@dynamic_prompt
def inject_retrieved_context(request: ModelRequest) -> str:
    """
    LLM이 호출되기 전에 무조건 벡터DB를 검색하고,
    검색 결과를 시스템 프롬프트에 강제로 주입한다.
    LLM은 검색을 스킵할 기회조차 없다.
    """

    # ① 사용자 질문 추출
    user_query = request.state["messages"][-1].text

    # ② 벡터DB 검색 (무조건 실행)
    docs = vector_store.similarity_search(user_query, k=3)
    context = "\n\n".join(d.page_content for d in docs)

    # ③ 검색 결과를 시스템 프롬프트로 반환 → LLM이 읽음
    return f"""너는 아래 컨텍스트만 보고 답변하라. 없으면 모른다고 해라.

컨텍스트:
{context}"""


# 미들웨어를 에이전트에 주입
agent = create_agent(model, tools=[], middleware=[inject_retrieved_context])

이 방식의 장점: 검색 100% 보장 + LLM 1회 호출. Tool Agent 방식(도구로 검색)은 LLM이 검색을 스킵할 수 있고 LLM 2회 호출이 필요함. 우리 파이프라인이 바로 이 Middleware Agent 패턴을 파이프라인 레벨에서 구현한 형태.

→ 우리 프로젝트 매핑: 핵심 설계 철학

우리는 LangChain의 Middleware를 그대로 쓰진 않았지만, 같은 원리로 동작함: src/rag/pipeline.py:154-167 — run_rag_chat() 가 run_rag_retrieve()를 먼저 호출해서 검색 결과를 확보한 뒤, 그 결과를 engine.answer()에 넘겨서 답변을 생성. LLM이 중간에 검색을 스킵할 수 있는 구조가 아예 존재하지 않음.

7. LangGraph — 직접 그래프 설계하기

이론: create_agent vs StateGraph

create_agent()는 편리하지만 내부 동작을 세밀하게 제어할 수 없음. "날씨 조회 전에는 무조건 날짜 확인을 거쳐라" 같은 규칙을 코드로 강제하려면 LangGraph의 StateGraph로 직접 노드와 엣지를 설계해야 함.

StateGraph의 3요소

요소설명비유
State그래프 전체에서 공유되는 데이터 구조. 노드가 입력으로 받고, 업데이트된 State를 반환작업 중인 공유 문서
Node하나의 작업 단위를 수행하는 파이썬 함수. State를 받아서 처리 후 새 State 반환작업자 한 명
Edge노드 간 연결. "일반 엣지"는 항상 다음 노드로, "조건부 엣지"는 State에 따라 분기작업 흐름도

예제: "날씨 조회 전 날짜 확인 강제" 그래프

from langgraph.graph import StateGraph, MessagesState, START, END
from typing import Literal, Annotated
import operator

# ① State 정의: 노드 간 공유할 데이터
class MyState(MessagesState):
    tool_hist: Annotated[list[str], operator.add]  # 사용한 도구 목록 (리스트 누적)

# ② Node: LLM 호출
def llm_call(state: MyState):
    response = model_with_tools.invoke(
        [SystemMessage("도구를 적절히 사용하라")] + state["messages"]
    )
    return {"messages": [response]}

# ③ Node: 도구 실행
def tool_node(state: MyState):
    results, names = [], []
    for tc in state["messages"][-1].tool_calls:
        tool = tools_by_name[tc["name"]]
        results.append(ToolMessage(content=tool.invoke(tc["args"]), tool_call_id=tc["id"]))
        names.append(tc["name"])
    return {"messages": results, "tool_hist": names}

# ④ Node: 날짜 확인 강제
def date_call(state: MyState):
    date_val = get_tomorrow_date.invoke({})
    return {"messages": [
        ToolMessage(content=f"날짜 확인 완료: {date_val}", tool_call_id="...")
    ]}

# ⑤ Routing: LLM 응답을 보고 어디로 갈지 결정
def conditional(state) -> Literal["tool_node", "date_call", END]:
    if not state["messages"][-1].tool_calls:
        return END                              # 더 쓸 도구 없음 → 종료
    calls = [tc["name"] for tc in state["messages"][-1].tool_calls]
    hist = state.get("tool_hist", [])
    if "get_weather" in calls and "get_tomorrow_date" not in hist:
        return "date_call"                     # 날짜 확인 강제!
    return "tool_node"                           # 정상 도구 실행

# ⑥ 그래프 조립
workflow = StateGraph(MyState)
workflow.add_node("llm_call", llm_call)
workflow.add_node("tool_node", tool_node)
workflow.add_node("date_call", date_call)

workflow.add_edge(START, "llm_call")                      # 시작 → LLM
workflow.add_conditional_edges("llm_call", conditional,     # LLM → 분기
    ["tool_node", "date_call", END])
workflow.add_edge("tool_node", "llm_call")               # 도구 → 다시 LLM (순환)
workflow.add_edge("date_call", "llm_call")               # 날짜확인 → 다시 LLM (순환)

app = workflow.compile()

실행 흐름 시각화

START→ llm_call→ conditional─→ tool_node→ 순환
└─→date_call→ 순환
└─→END

핵심: conditional()에서 날짜 미확인 시 date_call로 강제 분기. LLM이 이 규칙을 무시할 수 없음.

→ 우리 프로젝트 매핑: 미구현 (확장 포인트)
현재 파이프라인은 LangGraph 없이 선형 함수 호출로 동작하지만, pipeline.py의 run_rag_ingest() → run_rag_retrieve() → run_rag_chat() → run_rag_evaluation() 순서를 StateGraph의 노드로 감싸면 checkpoint/resume이 더 우아해지고, Multi-Query나 Self-RAG 같은 고도화 패턴을 노드 추가만으로 붙일 수 있음.

8. RAG 파이프라인 — 상세 작동 원리

전체 구조: 준비(Offline) + 질문(Online)

준비 단계 — 문서를 검색 가능하게 변환

1 원본 문서 읽기 → document_loader.py (PDF/DOCX/HWPX/TXT/MD 5종)
2 Text Splitter로 chunk 분할 → chunks.csv
3 Embedding 모델로 텍스트→벡터 변환 → embeddings.jsonl
4 Vector Store에 저장 → Chroma 또는 In-Memory

질문 단계 — 검색해서 답변 생성

1 사용자 질문 → Embedding으로 벡터 변환
2 Vector Store에서 similarity_search (k=3) → retrieval_results.jsonl
3 Context + 질문 + SystemMessage 조합 → LLM 호출 (또는 추출형 답변)
4 답변 + citation 생성 → answers.jsonl
5 평가: 정답과 비교 → metrics.json + 실패 분석 CSV들

Splitter — 문서를 왜, 어떻게 자르는가

파라미터의미너무 작으면너무 크면
chunk_size 한 chunk의 최대 글자 수 문맥 단절, 단답형 정보만 검색됨 검색 정밀도 저하, 관련 없는 내용 포함
chunk_overlap 앞뒤 chunk가 겹치는 글자 수 중복 증가, 저장 공간 낭비 chunk 경계에서 정보 잘림
(예: "총 예산은" / "50억원" 으로 찢어짐)

RFP 문서에선 chunk_size=500, overlap=80 정도가 일반적인 시작점. 우리 Config의 rag.splitter.chunk_size, rag.splitter.chunk_overlap이 이걸 제어.

Embedding — 텍스트를 숫자로 바꾸는 이유

"예산이 얼마야?"와 "총 예산은 50억원"이 얼마나 관련 있는지 계산하려면, 두 문장을 같은 공간의 숫자 벡터로 변환해야 함. Embedding 모델은 의미적으로 비슷한 문장을 벡터 공간에서 가까운 위치에 배치함. 코사인 유사도나 내적(dot product)으로 거리를 계산해서 가장 가까운 k개를 선택.

우리 Config의 rag.embedding.provider가 embedding 구현체를 결정: local = 의존성 없는 해시 기반 (빠름, 덜 정확), huggingface/openai = 실제 임베딩 모델 (정확, API/모델 필요).

검색 방법 3종 — 우리 retriever.py 기준

방식작동 원리장점단점Config 값
keyword 질문의 키워드가 chunk에 몇 개나 포함되는지 카운트 빠름, 의존성 없음, 정확한 용어 매칭 동의어/유의어 감지 못 함 retriever.method: keyword
semantic 질문과 chunk를 각각 벡터로 변환 후 코사인 유사도 계산 의미적 유사성 감지, 동의어 커버 임베딩 모델 필요, 키워드가 완전히 다르면 실패 가능 retriever.method: semantic
hybrid keyword 점수 × 0.4 + semantic 점수 × 0.6 으로 가중 평균 두 방식의 장점 결합, 가장 안정적 계산량 2배 retriever.method: hybrid

9. RAG 고도화 전략 (확장 로드맵)

왜 고도화가 필요한가 — 기본 RAG의 한계

한계증상고도화 전략
검색 표현 의존성 "입찰 조건"이라고 검색했는데 문서엔 "참가 자격"이라고 쓰여 있음 → 검색 실패 Multi-Query: 질문을 여러 표현으로 확장 후 병렬 검색

run_rag_ingest(config)
  ├── load_documents()          # → parsed_documents.csv
  ├── engine.chunk_documents()    # → chunks.csv (Splitter)
  └── engine.embed_chunks()       # → embeddings.jsonl (Embedding)

# ===== 2. Retrieve: 질문 → 관련 chunk 검색 =====
run_rag_retrieve(config, question)
  └── engine.retrieve()           # → retrieval_results.jsonl

# ===== 3. Chat: 검색 결과 → 답변 + citation =====
run_rag_chat(config, question)
  ├── run_rag_retrieve()       # 검색 결과 확보
  └── engine.answer()             # → answers.jsonl
       ├── local → build_answer()  # 추출형 (의존성 없음)
       └── ollama/openai → LLM     # 생성형 (Config provider 결정)

# ===== 4. Evaluate: 전체 평가 질문으로 지표 산출 =====
run_rag_evaluation(config)
  └── for each question:
       ├── run_rag_retrieve()
       ├── engine.answer()
       ├── metric 계산 (retrieval_hit, answer_match, citation_correct)
       └── _write_error_analysis()  # bad_retrievals, unsupported, failed
