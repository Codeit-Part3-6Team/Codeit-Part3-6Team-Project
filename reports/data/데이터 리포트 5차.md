오늘 진행한 작업 정리입니다. 리포트 4차의 "다음 단계"에 남겨둔 "`data_list_버전4.csv` 기준으로 RAG ingest 파이프라인을 재실행하고 `expected_chunk_id`를 채우는 작업"을 처리했습니다.

## 1. 요약

- `data_list_버전4.csv`(98개 문서)를 전용 ingest 경로로 청크화 → 16,218개 chunk 생성
- 과정에서 `engine: langchain` 설정이 실제로는 의존성 미설치로 단순 문자 분할(fallback)로 동작하고 있었다는 사실 확인
- `src/rag/document_loader.py`의 CSV 로더가 사업금액/발주기관 같은 메타데이터 컬럼을 청크 텍스트에서 누락시키는 버그를 발견 → 수정

- `chunks.csv`(16,218개) 자체의 무결성도 별도 점검 — 중복/누락/길이초과/인코딩노이즈/overlap 전부 정상 확인(8절)

## 2. 청킹 환경 구성

### 2-1. 입력 소스와 격리 폴더

원본 hwp/pdf 100개를 다시 파싱하지 않고, 리포트 4차에서 이미 재추출·검증을 마친 `data_list_버전4.csv`의 `텍스트` 컬럼을 그대로 ingest 대상으로 썼습니다. `src/rag/document_loader.py`에 이미 이 CSV 형식(`공고 번호`/`사업명`/`텍스트` 컬럼)을 읽는 `_parse_csv_document()`가 구현돼 있어서 새 파서 없이 바로 사용 가능했습니다.

`data/raw/` 폴더에는 v2·v3 버전 CSV와 `eval_questions*.csv`가 같이 있어서, ingest 로더가 폴더를 재귀적으로 전부 읽어버리면 의도치 않게 여러 버전이 같이 섞여 들어갑니다. 이를 막기 위해 **`data_list_버전4.csv` 단 하나만 symlink로 연결한 전용 폴더** `data/raw/v4_csv_only/`를 만들었습니다.

### 2-2. 전용 config

`configs/experiments/rag/rag_golden_v4.yaml`을 새로 만들어 위 폴더를 가리키게 했습니다. `paths.raw_docs_dir: data/raw/v4_csv_only`, `rag.loader.file_types: [csv]`, 출력은 `experiments/rag_golden_v4/`. 나머지 설정(embedding/retriever/answerer)은 기존 `rag_langchain.yaml`을 그대로 따라가서 다른 실험과 비교 가능하게 했습니다.

## 3. 청크 설정값 (chunk_size / chunk_overlap)과 선택 이유

**`chunk_size: 500`, `chunk_overlap: 80`** — 새로 튜닝한 값이 아니라 **기존 모든 config(`rag_langchain.yaml`, `rag_realistic_docs.yaml` 등)와 동일한 프로젝트 표준값을 그대로 재사용**했습니다.

이 값을 바꾸지 않은 이유는 `docs/team/pipeline_walkthrough.md`의 FAQ에 이미 근거가 있습니다: chunk_size를 200으로 줄이면 "제안 마감일은" / "7월 31일까지다"처럼 한 문장이 두 chunk로 쪼개져 검색이 실패하는 사례가 검증돼 있고, 500자 정도면 문장이 안 잘리고 한 chunk에 들어간다는 결론이 이미 있었습니다. 일관성을 유지하고 다른 실험과 비교 가능하게 하려고 그대로 따랐습니다.

step(다음 chunk가 시작되는 위치까지의 이동 거리) = chunk_size − chunk_overlap = **420자**입니다. 즉 chunk 1은 [0, 500), chunk 2는 [420, 920), chunk 3은 [840, 1340)... 식으로 80자씩 겹치며 진행합니다.

## 4. 실제 분할 로직 — `engine: langchain` 설정의 숨은 fallback

config는 `rag.engine: langchain`, `rag.splitter.type: recursive_character`로 돼 있어서 LangChain의 `RecursiveCharacterTextSplitter`(문단 → 줄 → 공백 순으로 끊을 자리를 찾는 방식)가 쓰일 것처�로 보이지만, **실제로는 그렇지 않았습니다.**

`src/rag/engines/langchain.py`의 `chunk_documents()`는 `langchain-core`/`langchain-text-splitters`를 import하려고 시도하고, 실패하면 조건이 맞을 때(embedding=local, answerer=local, vector_store=memory — 우리 config가 정확히 이 조건) `src/rag/chunker.py`의 단순 구현으로 **조용히 fallback**합니다. 이 작업 환경에는 `langchain_text_splitters`가 설치돼 있지 않아서 실제로 fallback이 동작했습니다. 증거로 chunk 경계를 까보면 단어/숫자 중간에서 그대로 잘립니다:

```
20241001798_chunk_0001 ...끝부분: "...□ 사업예산 : 130,"   ← 숫자 중간에서 끊김
```

실제로 동작한 `src/rag/chunker.py`의 로직은 매우 단순합니다 — 문장 경계를 전혀 신경 쓰지 않고 글자 수로만 자릅니다:

```python
def _split_text(text, chunk_size, overlap):
    clean_text = " ".join(text.split())   # 공백/줄바꿈을 전부 한 칸으로 압축
    start = 0
    while start < len(clean_text):
        end = min(start + chunk_size, len(clean_text))
        chunks.append(clean_text[start:end])
        if end == len(clean_text):
            break
        start = end - overlap
```

**팀이 알아야 할 재현성 이슈**: 누군가의 환경에 `langchain-text-splitters`가 설치돼 있다면, 같은 config·같은 문서로 ingest해도 실제 분할 결과(chunk 경계, chunk 수)가 달라질 수 있습니다. LangChain 쪽은 공백을 우선해서 끊으려 하기 때문입니다. 동일한 재현이 필요하면 `pip list | grep langchain`으로 두 환경의 설치 여부를 먼저 맞춰야 합니다.

## 5. document_loader.py 수정 — 메타데이터가 청크 밖으로 버려지던 문제

### 5-1. 증상

`expected_answer`가 "130,000,000원"처럼 `data_list_버전4.csv`의 `사업 금액` 컬럼 값 그대로인 질문 39건이 어떤 chunk에서도 매칭되지 않았습니다.

### 5-2. 원인

`_parse_csv_document()`가 원래 `텍스트` 컬럼만 chunk 대상 `text`로 사용하고, `사업 금액`/`발주 기관` 등은 `meta_*`라는 별도 필드로만 저장했습니다. 그런데 `chunker.py`의 `chunk_documents()`는 `document["text"]`만 자르고, `CHUNK_COLUMNS`(`pipeline.py`)에는 `meta_*` 필드가 아예 없어서 chunk 산출물에 복사조차 되지 않습니다. 즉 검증된 메타데이터 값이 **검색 대상(chunk text)에 한 번도 들어간 적이 없었습니다.**

추가로 39건 중 35건은 그 금액이 RFP 본문 어디에도 등장하지 않는다는 것도 확인했습니다(나라장터 등 외부 포털 메타데이터로만 존재 — 본문에는 안 적혀 있는 경우가 실제로 많습니다).

### 5-3. 수정 내용

`src/rag/document_loader.py`의 `_parse_csv_document()`가 각 문서 텍스트 맨 앞에 메타데이터 요약 줄을 붙이도록 변경했습니다:

```python
"text": _normalize_text(f"{_build_meta_preamble(entry, title)} {text}"),
# 결과 예시: "사업명: 한영대학교 ... 고도화 | 발주기관: 한영대학 | 사업금액: 130,000,000원 | 공고차수: 0.0 <본문...>"
```

`_build_meta_preamble()`은 `사업명`/`발주 기관`/`사업 금액`(천단위 콤마 + "원" 포맷)/`공고 차수`를 조합합니다.

**⚠️ 이건 `data_list_버전4.csv` 한정 작업이 아니라 CSV로 ingest하는 모든 경우에 영향을 주는 공용 코드 수정입니다.** 아직 커밋하지 않았으니 팀에 공유 후 리뷰·커밋 여부를 정해야 합니다.

### 5-4. 효과

재ingest 후 매칭 결과: 발주기관 79/98 → **98/98**, 예산 57/96 → **96/96**. (전체 합계는 6절 표 참고)

## 6. expected_chunk_ids 매핑 (chunks.csv → eval_questions_v4.csv)

### 6-1. 매핑 스크립트

`scripts/fill_expected_chunk_ids.py`를 새로 작성했습니다. 2단계로 매칭합니다.

**1차 — exact 매칭**: `expected_answer`를 줄바꿈(불릿) 또는 쉼표 목록으로 분리한 뒤, 각 항목이 같은 `source_doc_id`의 chunk text에 공백 제거 후 부분일치하는지 검사합니다. 모든 항목이 (서로 다른 chunk에 나뉘어 있어도) 발견되면 해당 chunk_id를 전부 기록합니다.

**2차 — fuzzy fallback**: exact 매칭이 실패하고 카테고리가 일정/자격/제출서류(LLM이 본문에서 의미 기반으로 추출한 카테고리)일 때만 동작합니다. `expected_answer`에서 키워드(한글 2글자 이상/영문/숫자)를 뽑아, 같은 문서의 chunk 중 키워드가 가장 많이 겹치는 chunk를 고릅니다. 토큰 1개짜리 답은 1개 이상, 그 외에는 2개 이상 겹쳐야 채택합니다. 이건 정확한 인용이 아니라 추정이므로 `note` 컬럼에 `[chunk_match: fuzzy, 키워드 N/M개 일치 — 검증 필요]`를 덧붙여 구분합니다.

### 6-2. 디버깅 — 처음 설계가 잘못됐던 부분과 수정

처음엔 fuzzy 검색 범위를 "LLM이 답을 추출할 때 실제로 본 입력(`generate_eval_questions_v4.py`의 `text[:3000]`)에 대응하는 앞쪽 chunk들"로 좁혔습니다. 이 가정으로 1차 개선했을 때 343건 중 3건이 미매칭으로 남았고, 그중 2건을 "LLM이 프롬프트의 예시 문구를 그대로 베낀 hallucination"으로 추정해 보고했습니다.

**원문을 직접 열어 대조한 결과 이 추정은 틀렸습니다.** 두 건 모두 실제 문서에 내용이 있었습니다:

| 문서 | 답변 | 실제 위치(글자 수) | 진짜 원인 |
|---|---|---|---|
| R25BK00564730 | 제안서, 사업자등록증, 실적증명서 | 369자 / 22,899자 / 23,210자 (3곳) | 쉼표로 나열된 항목이 본문 곳곳에 흩어져 있었는데, 매칭 로직이 쉼표를 줄 구분자로 인식하지 못함 |
| 20240345257 | 소프트웨어사업자 등록, 최근 3년 실적 보유 | 47,503자 | 자격조항이 본문 뒤쪽(별지서식 부근)에 있어 "처음 3000자 안에 답이 있다"는 검색범위 제한 가정에 걸려 제외됨 |

즉 RFP는 자격/제출서류 조항이 본문 훨씬 뒤쪽 별지서식에 있는 경우가 많아서, "LLM이 본 입력 = 정답 위치"라는 가정 자체가 틀렸습니다. 이를 반영해 ① fuzzy 검색 범위 제한을 제거(문서 전체에서 탐색), ② exact 매칭에 쉼표 목록도 인식하도록 추가 → 343/344까지 회복했습니다.

마지막 1건(`R25BK00601569`, "계약기간: ~2025년")도 직접 원문 대조로 `..._chunk_0009`("나. 용역기간 : 계약체결일로부터 2025.12.20일까지")가 근거임을 확인하고 수동으로 채워 **최종 344/344**를 완료했습니다.

### 6-3. 추가 개선 — 인용부호 처리

fuzzy 54건 중 일정 카테고리 7건("'착수일로부터 60일'"처럼 LLM이 답을 따옴표로 감싼 경우)을 직접 보니, **따옴표만 벗기면 원문과 완전히 일치하는 진짜 exact 매칭 건**이었습니다. `exact_match()`가 줄/쉼표 경계의 `-•‣▪` 같은 불릿 문자는 벗기면서 양끝 따옴표(`'` `"` `"` `"`)는 안 벗기고 있던 게 원인이었습니다. 이를 고쳐 exact 296건 + fuzzy 47건(미매칭 1건은 6-2와 동일하게 수동 처리)으로 다시 줄였습니다.

### 6-5. fuzzy 매칭 검증 결과

`scripts/verify_fuzzy_chunk_matches.py`를 새로 작성해 fuzzy 47건을 자동으로 1차 검증했습니다. `expected_answer`를 항목 단위로 나눠 각 항목이 ① 배정된 chunk 안에 있는지, ② 같은 문서의 다른 chunk에 있는지(있으면 잘못 배정된 것), ③ 문서 전체에 없는지를 확인합니다(항목이 1개뿐인 단답은 답에 들어있는 숫자가 배정된 chunk에 있는지로 보조 판정).

**자동 검증 결과**: PARTIAL(항목 일부만 확인) 41건, WEAK(단답+숫자 불일치/숫자 없음) 4건, PLAUSIBLE(단답+숫자 전부 일치) 2건. 카테고리별로 신뢰도가 뚜렷하게 갈렸습니다 — **제출서류는 항목 평균 82% 확인**(원문 그대로 나열한 목록이라 검증이 잘 됨), **자격은 항목 평균 9% 확인**(LLM이 원문을 그대로 인용하지 않고 자기 표현으로 재구성해서 항목 단위 대조가 거의 안 통함).

자동 검증에서 WEAK로 나온 4건 + 자격 카테고리 중 항목이 0% 확인된 표본을 **직접 원문을 열어 대조**했고, 그 결과 다음 7건을 수동으로 정정/확인했습니다.

## 8. 청킹 최종 점검

매핑/검증 작업을 마치고, `chunks.csv`(16,218행) 자체가 깨지지 않고 제대로 만들어졌는지 별도로 점검했습니다.

| 점검 항목 | 결과 |
|---|---|
| `chunk_id` 중복 | 0건 |
| 문서 커버리지 (`data_list_버전4.csv` 98건 ↔ `chunks.csv`에 등장한 문서) | 누락 0건, 알 수 없는 문서 0건 — 98/98 일치 |
| 문서별 chunk 번호 연속성(`_chunk_0001`부터 빠짐없이) | 깨진 문서 0건 |
| chunk 길이 범위(0~500자) | 500자 초과 0건, 빈 chunk 0건 (최소 90자, 최대 500자, 평균 498.3자) |
| 인코딩 노이즈(제어문자·replacement문자 `�`) | 0건 |
| `token_count` 필드 정합성 | 비정상(0 또는 숫자 아님) 0건 |
| 인접 chunk 간 overlap(설정값 80자) | 16,120개 인접쌍 전부 실제 overlap 78~80자 — 70자 미만(내용 손실 의심) 0건 |
| 문서 전체 커버리지(끝까지 빠짐없이 chunk화됐는지) | 불일치 0건 |

overlap 항목에서 처음엔 "80자 정확히 일치"만 봤을 때 16,120쌍 중 6,742쌍이 어긋나 보였지만, 직접 까보니 `chunker.py`의 `_split_text()`가 각 chunk 끝에 `.strip()`을 적용해서 공백이 걸리면 78~79자로 1~2자 줄어드는 것뿐이었습니다(실제 겹치는 글자 수 분포: 80자 9,378쌍, 79자 5,811쌍, 78자 931쌍 — 전부 70자 이상). **내용이 실제로 빠진 경우는 0건**으로, 청킹 자체는 정상입니다.

## 9. 한계 및 다음 단계

- **fuzzy 매칭 41건 중 자격 18건/제출서류 21건은 완전히 검증되지 않았습니다.** 6-5절에서 직접 원문 대조로 7건은 교정·확인했지만, 나머지는 "문서에 같은 단어가 있다"까지만 자동 확인됐고 "정확히 그 조항"인지는 사람이 봐야 합니다. 특히 자격 카테고리는 답변이 LLM의 재구성 문장이라 항목 단위 대조 자체가 잘 안 통합니다. `eval_questions_v4_fuzzy_review.csv`를 일치율 낮은 순으로 일부 더 대조하는 걸 권장합니다.
- **"목차(TOC) chunk 오배정" 패턴이 한 번 더 발견됐습니다(6-5절).** chunk_0001~0004처럼 표지/목차에 가까운 chunk가 제목 단어만 겹쳐서 잘못 선택되는 경우가 있었습니다. 이미 발견한 4건은 고쳤지만, 비슷한 패턴이 남아있을 수 있어 fuzzy 검토 시 "배정된 chunk_id가 유난히 작은 번호인지"를 같이 보면 좋습니다.
- **`src/rag/document_loader.py` 수정이 아직 미커밋 상태입니다.** 같은 파일에 리포트 4차에서 언급된 HWP 파서 수정사항도 같이 미커밋으로 남아있어, 커밋 전에 두 변경을 함께 리뷰해야 합니다.
- **재현성 이슈**: `langchain-text-splitters` 설치 여부에 따라 같은 config로도 실제 청크 경계가 달라질 수 있습니다(4절 참고). 팀원 간 환경을 맞추거나, 결과를 비교할 때 이 점을 감안해야 합니다.
- 리포트 4차에서 남겨둔 "RAG 성능 측정 미실시" 항목은 여전히 미완입니다. 이제 `chunks.csv` + `embeddings.jsonl` + `expected_chunk_ids`가 모두 준비됐으므로, `scripts/run_rag_retrieve.py` / `run_rag_chat.py --evaluate`로 실제 retrieval recall·answer accuracy를 측정하는 게 다음 단계입니다.
