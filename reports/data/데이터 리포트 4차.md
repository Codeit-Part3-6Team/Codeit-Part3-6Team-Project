## 4. 텍스트 컬럼 대규모 추출 누락 발견 및 재추출

### 4-1. 발견 경위

결측치/이상치를 점검하던 중 텍스트 길이가 500~1500자에 불과한 행이 21건 발견됐습니다. 내용을 열어보니 전부 "주주평등원칙(상", "[별첨 3] 소프트웨어"처럼 단어·목차 항목 중간에서 갑자기 끊겨 있었습니다. 원본 hwp 파일 크기(예: (사)벤처기업협회 4.2MB)와 추출된 텍스트 길이(503자)를 비교해보니 명백히 추출 실패였습니다.

### 4-2. 원인

리포트 2차에서 이미 같은 유형의 문제(텍스트 500자 미만 7건)를 발견해 HWP5 레코드 기반 파서로 수동 복원한 적이 있었습니다. 문제는 **그 수정이 나머지 91건(hwp 88건 + pdf 3건)에는 적용되지 않은 상태로 남아있었다는 것**입니다. `src/rag/document_loader.py`에 같은 방식(BodyText 레코드 중 `HWPTAG_PARA_TEXT`(tag_id=67)만 디코딩)의 커밋되지 않은 수정사항이 이미 작성돼 있었으나, `data_list_버전3.csv`에는 반영되지 않은 상태였습니다.

### 4-3. 검증 후 재추출

표본(랜덤 5개 hwp)으로 새 파서와 기존 텍스트를 비교한 결과 평균 32배 더 많은 본문이 나왔고, PDF 3건도 마찬가지였습니다(예: 고려대학교 297페이지 문서가 기존엔 1페이지만 추출됨). 글자 단위 prefix 대조로 새 텍스트가 기존 텍스트의 "이어지는 연속"임을 확인한 뒤(요약/키워드 추출이 아니라 같은 추출이 더 길게 이어진 것), `scripts/reextract_all_text_v3.py`를 작성해 hwp/pdf 전체 98행에 적용했습니다 (기존 텍스트가 더 짧거나 같은 7건은 그대로 유지, 나머지 91건만 교체).

## 6. eval_questions_v4.csv 생성 (골든데이터셋)

### 6-1. 생성 방식

`scripts/generate_eval_questions_v4.py`(기존 v3 스크립트 기반)로 `data_list_버전4.csv`를 입력해 생성했습니다.
- 예산/발주기관: CSV 컬럼값을 그대로 사용(LLM 미사용, hallucination 위험 없음)
- 일정/자격/제출서류: OpenAI gpt-4o-mini(temperature=0)로 문서 본문에서 의미 기반 추출, 본문에 없으면 질문 생성 제외

### 6-2. 생성 중 발견하고 수정한 버그

1. **"정보없음" 필터 누락**: LLM이 "입찰 참가 자격: 정보 없음"처럼 라벨을 붙여 응답하면 기존의 정확히-일치 검사(`!= "정보 없음"`)를 통과해버려 무의미한 답변이 섞여 들어갔습니다. 짧은 응답에서 "정보없음"이 포함되면 걸러내도록 수정 → 제출서류 11→55건, 자격 7→29건(중복 제거 전 기준)으로 정상 회복
2. **내용 중복 제거 임계값 오류**: 카테고리 내 답변 유사도로 중복을 제거하는 로직에서, "계약일로부터 6개월"과 "계약일로부터 3개월"처럼 **숫자만 다른 답변**이 문구 유사도(0.85) 때문에 같은 것으로 잘못 묶여 일정 카테고리 93→58건으로 과도하게 줄어드는 문제를 발견. 답변에서 추출한 숫자가 다르면 절대 중복으로 보지 않도록 보강(임계값도 0.92로 상향) → 93→84건으로 안전하게 보존
3. **비공개 예산 오답**: 을지대학교/보험개발원처럼 사업금액이 "비공개"라 0/1원으로 채워진 2건이 예산 카테고리에 "0원"/"1원"이라는 잘못된 정답으로 들어간 것을 발견해 제거. 스크립트에도 `금액 >= 1000` 조건을 추가해 재발 방지

## 7. 작업 근거 Q&A

작업 검증 과정에서 받은 질문과 답변을 근거 자료로 정리합니다.

### Q1. 문서 길이가 32배 증가했는데, 그게 실제 정보량 증가인지 어떻게 검증했나?

세 가지 방법으로 교차 검증했습니다.

- **글자 단위 prefix 대조**: 기존 텍스트(공백 정규화)가 새 텍스트의 시작 부분과 글자 단위로 거의 100% 일치하는지 확인했습니다. 예를 들어 GKL 그룹웨어 문서는 기존 텍스트 전체가 새 텍스트의 앞부분과 그대로 일치했고, 다른 샘플들도 따옴표·글머리 기호 같은 사소한 디코딩 차이 외에는 한 글자도 다르지 않았습니다. 즉 새 텍스트는 "다른 추출"이 아니라 기존에 끊겼던 지점에서 "이어서 계속 읽은 것"입니다.
- **끊기는 지점의 의미 검토**: 기존 텍스트가 끊긴 지점을 직접 열어보면 "주주평등원칙(상"(단어 중간), "경기도 버스정보시스템"은(문장 중간), "[별첨 3] 소프트웨어"(목차 항목 중간)처럼 의미상 아무 의도 없이 임의로 끊겨 있었습니다. 요약이나 키워드 추출이었다면 이런 식으로 끊기지 않습니다.
- **노이즈 검사**: 새로 추출한 텍스트 91건 전체에 제어문자·유니코드 replacement 문자(�)·lone surrogate가 0건이었고, 문서 끝부분이 서명란·제출서식 목록처럼 자연스럽게 종료됐습니다(4-4절 표 참고).

### Q2. 새 파서가 올바르게 추출했다는 근거는?

- **포맷 스펙 기반 파싱**: HWP5는 BodyText 섹션이 4바이트 레코드 헤더(tag_id+size)로 구성되고, tag_id=67(HWPTAG_PARA_TEXT)만 실제 문단 텍스트입니다. 새 파서는 이 레코드 타입만 골라서 디코딩하므로 표/그림/스타일 레코드의 바이너리가 섞일 수 없는 구조입니다(기존 vendor 추출은 레코드 구분 없이 통째로 디코딩해서 이 노이즈가 본문 사이에 끼어든 게 원인 — 4-2절 참고).
- **독립적인 사전 검증과의 일치**: 리포트 2차에서 이미 같은 방식으로 7개 문서를 수동 복원해 인간이 직접 검토한 적이 있는데, 오늘 새 파서로 이 7건을 다시 돌려보니 거의 동일한 결과(공백 정규화 차이 3~6% 내)가 나왔습니다 — 같은 방법론이 재현된다는 뜻입니다.
- **PDF는 별개 라이브러리(pypdf)로 페이지 전체를 순회**해서 같은 현상(예: 고려대 297페이지 중 1페이지만 추출되던 것)을 확인했고, 이는 HWP 파서와 무관한 별도 검증입니다.
- 기존 테스트 스위트 통과(`pytest tests/`).

### Q3. 파싱 개선으로 RAG 성능이 얼마나 향상됐나? — **아직 측정하지 않음**

오늘 한 작업은 (1) 메타데이터 결측치 보완, (2) 텍스트 추출 완전성 개선, (3) 골든 데이터셋 재생성까지입니다. **실제 RAG 파이프라인을 돌려 retrieval/answer 품질을 old vs new로 비교 측정하지는 않았습니다.** 측정하려면 ① 기존 corpus(텍스트 일부만 있던 버전)와 `data_list_버전4.csv`로 각각 RAG ingest 실행 → ② `eval_questions_v4.csv` 344문항으로 두 인덱스에 대해 retrieval recall/answer accuracy를 비교해야 합니다(7절 다음 단계와 동일).

### Q4. 골든 데이터셋 생성 시 LLM Hallucination을 어떻게 방지했나?

- **예산/발주기관(98+96=194문항, 전체의 56%)은 LLM이 답을 생성하지 않습니다.** CSV의 `사업 금액`/`발주 기관` 컬럼 값을 그대로 정답으로 쓰고 질문도 고정 템플릿이라 hallucination 발생 지점이 없습니다.
- **일정/자격/제출서류(150문항)는 LLM(gpt-4o-mini, temperature=0)을 쓰지만 프롬프트로 제약**했습니다: "문서에 없는 내용은 추측하지 말 것", "문서에 없으면 정확히 '정보없음'이라고만 답할 것"을 명시했고, 그렇게 나온 "정보없음" 답변은 최종 데이터셋에서 전부 제외했습니다(6-2절 버그 수정 참고).
- 이 안전장치를 실제로 검증하기 위해 **25개 샘플을 무작위 추출해 원문과 직접 대조**했고(6-4절), hallucination 0건을 확인했습니다. 다만 344개 전체를 사람이 한 줄씩 대조한 전수 검증은 아니라는 점은 한계로 남습니다.
